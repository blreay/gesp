# /gesp-mock-exam — GESP C++ 模拟试卷生成器

根据 CCF GESP 考纲知识点文档和历年真题，分析出题趋势和高频考点，批量生成交互式 HTML 模拟试卷。

## 能力

- 📄 读取并分析知识点文档（PDF/文本文件/目录）
- 📝 读取并分析历年真题（PDF 目录）
- 🎯 按 GESP 等级（1-8 级）匹配题型和难度
- 🏭 批量生成模拟试卷，每套包含：
  - **交互式 HTML 试卷**：可点击选择、自动判卷、错题高亮、一键复制到错题本
  - **HTML 参考答案**：含详细解析
  - **C++ 测试程序**：每道编程题一个，自动评测考生代码
- 📑 生成 index.html 首页索引所有试卷

---

## Step 0 — 收集输入参数

使用 AskUserQuestion 收集 4 个必要参数。如果用户在调用时已经通过 args 提供了部分参数，跳过已提供的参数。

需要收集的参数：
1. **知识点文档路径** — PDF 文件或包含知识点文件的目录
2. **真题目录路径** — 包含历年真题 PDF 文件的目录
3. **GESP 等级** — 1 到 8 级
4. **试卷数量** — 要生成几套模拟试卷

如果用户没有通过 args 提供完整参数，用 AskUserQuestion 逐项收集：

```
请提供以下信息：
1. 知识点文档路径（PDF文件或目录）
2. 真题目录路径（包含真题PDF的目录）
3. GESP等级（1-8）
4. 需要生成的试卷数量
```

收集完成后，验证路径存在：
```bash
ls -la <知识点路径>
ls -la <真题目录>/*.pdf
```

如果路径不存在，提示用户修正。

## Step 1 — 分析知识点和真题

### 1.1 读取知识点文档
- 如果是 PDF 文件，用 Read 工具读取（支持分页）
- 如果是目录，遍历目录内所有 PDF/文本文件逐一读取
- 提取知识点列表、考纲要求、难度分级

### 1.2 读取所有真题
- 遍历真题目录，对每个 PDF 文件用 Read 工具读取
- 分析每套真题的：
  - **题型结构**：选择题数量/分值、判断题数量/分值、编程题数量/分值
  - **知识点分布**：每道题考察的知识点
  - **难度水平**：基础/中等/较难
  - **出题模式**：代码跟踪/表达式求值/填空/概念判断等

### 1.3 生成分析报告
输出一份简要的分析摘要，包括：
- 题型固定结构（如：15单选+10判断+2编程）
- 高频知识点排名（按出现频次）
- 编程题常见主题
- 难度趋势

## Step 2 — 规划试卷内容

根据分析结果，为每套试卷规划：
- 知识点覆盖分配（确保不同试卷间有差异化，同时覆盖高频考点）
- 编程题主题选择（不与真题重复，难度递进）
- 每套试卷的侧重点标签（用于 index 页展示）

规划原则：
- 第 1-2 套偏基础，第 3-5 套中等，第 6-8 套中等偏上，第 9-10 套综合冲刺
- 高频考点（如循环跟踪、运算符优先级、变量命名）每套都要覆盖
- 编程题 20 套不重复，涵盖：时间计算、数位操作、因数/质数、累加求和、条件统计等

## Step 3 — 批量生成试卷

**重要：每次只让一个 agent 生成 2 套试卷，避免 token 超限。**

对于 N 套试卷，启动 ceil(N/2) 个并行 agent，每个 agent 负责 2 套（最后一个可能只 1 套）。

每个 agent 的 prompt 模板：

```
你需要在 <output_dir> 下生成第 X 套和第 Y 套 GESP C++ <level>级模拟试卷。

## 题型结构
<从 Step 1 分析得出的题型结构>

## 难度要求
<该套的难度定位>

## 第 X 套
单选和判断题重点考：<知识点列表>
编程题：
1. 「<题目名>」— <简要描述>
2. 「<题目名>」— <简要描述>

## 第 Y 套
<同上>

## 每套需要生成的文件（4个文件）：
1. exam_XX/mock_exam_XX.html — 试卷
2. exam_XX/mock_exam_XX_answers.html — 答案与解析
3. exam_XX/test_XX_prog1.cpp — 编程题1的测试程序
4. exam_XX/test_XX_prog2.cpp — 编程题2的测试程序

## HTML 试卷要求
- 美观的 CSS 排版（渐变标题栏、代码高亮、卡片式题目）
- 分值表格

## 答案 HTML 要求
- 答案汇总表格
- 每题详细解析
- 编程题参考代码

## 测试程序要求
- 接受命令行参数（考生程序路径）
- 用临时文件传输入，捕获输出
- 至少 10 组测试数据
- 表格展示失败用例

## 验证步骤
生成完后，编译测试程序和参考答案，运行测试确认全部通过。
```

等待所有 agent 完成。

## Step 4 — 添加交互功能

生成完所有试卷后，需要给每套试卷的 HTML 添加交互式判卷功能。

### 4.1 首先手动改造第 1 套试卷
读取第 1 套试卷和答案，为试卷 HTML 添加以下交互功能：

**CSS 增强：**
- 选项可点击：`.opt` 带 `radio-dot` 圆形按钮，选中高亮
- 判断题：`.tf-opt` 药丸形按钮
- 判卷后：`.wrong` 红色背景 + `.correct-graded` 绿色背景
- `.correct-answer-tag` 正确答案标签
- `.btn-copy` 复制按钮
- `.modal-overlay` 未完成提示弹窗
- `.toast` 复制成功提示

**HTML 结构改动：**
- header 加判卷按钮 `btn-grade`
- 每道题加 `data-qtype`, `data-qnum`, `data-answer` 属性
- 选项改为可点击 div，带 `onclick="selectOpt(this)"`
- 判断题选项改为药丸按钮 `onclick="selectTF(this)"`
- 每题加 `btn-copy` 和 `correct-answer-tag`
- 分数表格得分行加 id 和 scored class

**JavaScript：**
- `selectOpt(el)` — 单选选中逻辑
- `selectTF(el)` — 判断题选中逻辑
- `gradeExam()` — 判卷主逻辑：检查未完成→弹窗提示→计分→高亮对错→显示分数
- `copyQuestion(btn)` — 复制错题到剪贴板（含题目、选项、错误答案、正确答案）
- `showToast(msg)` — 提示信息

### 4.2 编写 Python 转换脚本
以第 1 套改好的交互版 HTML 为模板，编写 `convert_to_interactive.py` 脚本，自动把剩余试卷批量转换为交互版。

脚本逻辑：
1. 从第 1 套交互版 HTML 中提取 CSS 块、JavaScript 块、modal/toast HTML
2. 对每套非交互版试卷：
   - 从 answers.html 提取正确答案（单选 + 判断）
   - 替换 CSS
   - 改造 header（加判卷按钮）
   - 改造单选题（加 data 属性、radio-dot、onclick）
   - 改造判断题（加 data 属性、药丸按钮）
   - 注入 modal、toast、JavaScript
3. 写回文件

```bash
python3 convert_to_interactive.py
```

### 4.3 验证所有试卷
```bash
for i in $(seq -w 1 N); do
    f=exam_$i/mock_exam_$i.html
    grep -c "gradeExam" "$f"      # 应大于 0
    grep -o 'data-answer=' "$f" | wc -l  # 应等于总题数
done
```

## Step 5 — 生成 index.html

生成一个首页索引，包含：
- 深色渐变背景，科技感设计
- 顶部统计：N 套卷 / M 道选择判断 / K 道编程
- 每套试卷一张卡片，包含：
  - 彩色渐变编号
  - 试卷标题和副标题
  - 知识点标签
  - 编程题名称
  - "开始答题 →" 按钮
- 全部使用相对路径链接

## GESP 等级题型参考

### 一级
- 15 道单选（2分×15=30分）+ 10 道判断（2分×10=20分）+ 2 道编程（25分×2=50分）
- 知识范围：顺序结构、选择结构（if-else）、循环结构（for/while/do-while）、基本数据类型、输入输出、运算符、计算机常识

### 二级
- 15 道单选 + 10 道判断 + 2 道编程
- 增加：数组（一维）、字符串基础、函数基础、简单排序

### 三级及以上
- 题型可能变化，请根据真题分析确定

---

## 注意事项

1. **答案正确性**：所有计算题必须验证，代码跟踪题手动推演
2. **测试程序**：必须编译通过且参考答案全部通过测试
3. **不重复**：各套试卷之间题目不重复，编程题主题不重复
4. **交互功能**：所有试卷必须具备完整交互功能（判卷、错题高亮、复制）
5. **并行生成**：利用 Agent 工具并行生成，但每个 agent 最多处理 2 套避免 token 超限
